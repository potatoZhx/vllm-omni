## 二、根本差异说明

### 2.1 成本估算路径不同（所有偏差的首要来源）

仿真与实测在调度排序时使用的 **estimated_cost_s** 值不同：

| 请求类型 | 实测 estimated_cost_s | 仿真 profile 值 | 偏差 |
|----------|----------------------|-----------------|------|
| 512×512, 20步 | 3.2334 | 2.504 | +29.1% |
| 768×768, 20步 | 5.2415 | 4.793 | +9.4% |
| 1024×1024, 25步 | 11.5419 | 11.071 | +4.3% |
| 1536×1536, 35步 | 37.7646 | 35.912 | +5.2% |

- **实测系统**：未配置 `--diffusion-runtime-profile`。Benchmark 客户端先执行 8 个 1-step warmup 请求，测出 base_time，然后按 `area × frames × steps` 缩放得到 `estimated_cost_s`，注入请求的 `extra_args`。调度器优先使用该注入值。
- **仿真器**：使用 `qwen_A100.json` 中精确的纯硬件执行时间作为成本估算（查表法）。

> **为什么仿真不做启发式/warmup？** 仿真无法执行真实 GPU 推理，无 warmup 数据可依赖。查表法在独立仿真时可稳定复现，而 warmup 依赖运行时环境。这是已知且合理的差异。

### 2.2 硬件执行时间方差

仿真使用恒定的 profile 时间；实测存在 ±0.5s 的方差（GPU 温度、调度抖动等）。高负载下多请求竞争，微小时间差异改变队列状态，级联放大为排序分歧。

### 2.3 service_time / waiting_time 统计口径差异

抢占算法下，实测 service_time = `finish - first_dispatch`（挂钟时间，含被抢占后空等），仿真 service_time = `sum(chunk_service_times)`（纯 GPU 计算时间）。因此抢占场景下两者的 service_time / waiting_time 数值不可直接对比，但 **latency = waiting_time + service_time** 两边定义一致，可直接对比。

### 3.2 sjf_aging

公式：`aged_cost = estimated_cost / (1 + aging_factor × cost_weight × age)`，其中 `cost_weight = clamp(sqrt(cost / 12.0), 1.0, 4.0)`。

排序键：`(aged_cost, estimated_cost, enqueue_time)`

仿真实现与 `stage1_scheduler._build_sjf_aging_queue` 完全一致。

#### rps=0.05

| service_time_mean | 11.25 | 11.30 | -0.40% |

**完成顺序**: 100/100 (100%) — 完全对齐

低负载下队列几乎不积压，sjf_aging 退化为类 FCFS，排序决策极少触发。

**完成顺序**: 92/100 (92%) — 首次不匹配 index=24

**P95 偏差 +6.90% 解释**: latency_mean 仅差 +0.22%，说明整体排序质量良好。P95 偏差来自分位数对排序微调的敏感性——8 处不匹配导致若干 1536×1536 请求的完成顺序略有调整，恰好影响了 P95 所在区间。P99（+0.24%）和 mean 均无问题，说明偏差集中在 P95 附近的少数请求，而非系统性错误。

8 处顺序不匹配均为成本估算差异导致的 aged_cost 微小翻转：仿真中 512×512 的 cost_weight=1.0（cost=2.504, 2.504/12=0.209, sqrt=0.457, clamp→1.0），实测中 512×512 的 cost_weight=1.0（cost=3.234, 3.234/12=0.270, sqrt=0.519, clamp→1.0），虽然 weight 相同，但分子 estimated_cost 不同（2.504 vs 3.234），改变了 aged_cost 的相对大小。

#### rps=0.125

| 指标 | 仿真 | 实测 | 偏差% |
|------|------|------|-------|
| duration | 1134.31 | 1132.14 | +0.19% |
| throughput_qps | 0.0882 | 0.0883 | -0.11% |
| latency_mean | 85.72 | 78.50 | **+9.19%** |
| latency_p95 | 295.48 | 312.52 | -5.45% |
| latency_p99 | 338.61 | 339.80 | -0.35% |
| waiting_time_mean | 74.47 | 67.16 | +10.88% |
| service_time_mean | 11.25 | 11.34 | -0.78% |

**完成顺序**: 62/100 (62%) — 首次不匹配 index=7

**+9.19% 偏差解释**: 高负载下 load factor > 1（请求到达速率超过服务能力），队列持续积压。aging 排序每次重新评估时，所有排队请求的 aged_cost 都参与竞争。

核心影响链：
1. 成本估算差异使 512×512 的仿真 aged_cost 更低（2.504 vs 3.234），更容易"抢到"优先权
2. 仿真中小请求被更积极地优先处理 → 大请求等待更久 → 大请求 age 增长 → 最终通过 aging 获得优先权，但时间点已偏移
3. 每次排序偏差影响后续请求的 age 和竞争格局，在 100 个请求中级联放大
4. 最终表现为仿真 latency_mean 偏高（小请求更快但大请求更慢，大请求权重大），P95 反而偏低（大请求通过 aging 最终得到执行，但整体延迟分布形状不同）

P95 的反向偏差（-5.45%）正好说明了这一点：仿真中大请求的延迟分布右尾更短（aging 最终将它们提升），但中位数区域延迟更高。

### 3.3 sjf_aging_guarded

与 sjf_aging 相同的 aging 公式，额外增加尾部保护：`wait_time > protection_threshold_s` 时强制提升到队首。使用更小的 chunk budget（5 步）和更低的小请求阈值（6000ms），因此 1024×1024 也被 chunk 化。

#### rps=0.125

| 指标 | 仿真 | 实测 | 偏差% |
|------|------|------|-------|
| duration | 1134.31 | 1133.15 | +0.10% |
| throughput_qps | 0.0882 | 0.0882 | 0.00% |
| latency_mean | 123.53 | 122.86 | +0.55% |
| latency_p95 | 268.57 | 267.14 | +0.53% |
| latency_p99 | 314.68 | 313.03 | +0.53% |
| waiting_time_mean | 112.27 | 110.83 | +1.30% |
| service_time_mean | 11.25 | 12.02 | -6.38% |

**完成顺序**: 88/100 (88%)

**对齐优秀 (+0.55%) 的原因**:
1. **更频繁的 chunk 重新评估**：budget=5 使 1024×1024 被分为 3 次调度（5+5+10 步），1536×1536 分为 7 次调度。每 ~2.2s 重新评估一次排序，快速修正任何因成本估算差异导致的排序偏差。
2. **尾部保护机制**：等待过久的请求被无条件提升，降低了成本估算对最终调度结果的影响。
3. **service_time -6.38%**：口径差异（挂钟 vs 纯 GPU），预期行为。latency 对齐良好说明整体调度等价。

### 3.8 p95_bucket_sjf

按 learned P95 绝对延迟分桶（4 个桶），桶内按 estimated_cost SJF 排序。

桶分配：`target_p95 = max(history_p95, cost_ms)`，`deadline = arrival + target_p95/1000`，`urgency = (deadline - now) × 1000`，`bucket = urgency / bucket_width`。

#### rps=0.05

**+7.38% 偏差解释**:

p95_bucket_sjf 的桶分配依赖两个成本相关量：
1. `max_estimated_cost_ms`：影响 `anchor_window_ms` 和 `bucket_width_ms`
2. `target_p95_ms = max(history_p95, cost_ms)`：影响每个请求的 deadline

成本估算差异同时影响这两个量。以 512×512 为例：
- 仿真 cost_ms = 2504，实测 cost_ms = 3234
- 仿真中 512×512 的 `target_p95 = max(history, 2504)`
- 实测中 512×512 的 `target_p95 = max(history, 3234)`

当 history_p95 < 3234 时，实测给 512×512 更宽松的 deadline（+29%），使其看起来更不紧迫 → 放入更高编号的桶 → 更靠后执行。仿真中 512 的 deadline 更紧 → 更早执行。

这解释了 median 偏差 +11.46%：仿真中中等请求等待更久（因为小请求被更积极地优先处理）。而 P95/P99 偏差很小（<2.5%），因为大请求（1536×1536）的 cost 偏差仅 5%，bucket 分配差异小。

waiting_time +26.48% 和 service_time -6.81% 的巨大差异来自口径差异（仿真统计纯 GPU 时间，实测挂钟时间含抢占空等）。

经逐函数与 `stage1_scheduler.py` 对照核查：

**已知的微小实现差异**（不影响实际结果）：
- 排序键的 tie-breaker 使用 `arrival_time` 而非 `enqueue_time`（仅在主键完全相同时生效，实测中概率极低）
- 排序键末位为字符串 `request_id` 而非整数 `sequence_id`（同理，仅在所有前序键相同时生效）

这些差异在当前基准测试（100 请求、4 种尺寸）中不会产生可观测影响。

## 五、结论

1. **仿真逻辑正确**：所有调度算法的排序公式、chunk 抢占、P95 学习均与真实调度器对齐，无逻辑 bug。

2. **偏差分级**：
   - **< 1%**: fcfs(中高负载)、sjf_aging(低负载)、sjf_aging_guarded、sjf_aging_no_preempt(低中负载)、size_bucket_sjf_aging(低负载)、size_bucket_sjf_aging_no_preempt、p95_first(全部)、p95_bucket_sjf_norm(高负载)
   - **1~5%**: p95_bucket_sjf(低负载)、p95_bucket_sjf_norm(低中负载)、size_bucket_sjf_aging(中高负载)
   - **5~10%**: sjf_aging(高负载)、sjf_aging_no_preempt(高负载)、p95_bucket_sjf(中高负载)
   - **>10%**: fcfs rps=0.05（已知：低负载未建模系统开销）

3. **偏差根因汇总**：

| 根因 | 影响范围 | 说明 |
|------|---------|------|
| 成本估算差异 (warmup vs profile) | sjf_aging 系列 + p95_bucket_sjf 高负载 | 512×512 偏差 +29% 最大，级联放大 |
| 在线学习收敛差异 | p95 系列早期 | 前几个 chunk 的时间差异导致 EMA 收敛路径不同 |
| 硬件执行时间方差 | 所有算法微量 | ±0.5s 波动，低负载表现为 < 1% 偏差 |
| 未建模系统开销 | 仅 fcfs rps=0.05 | GPU context switch、VAE slicing 等 |
| service_time 口径差异 | 抢占算法的 service/waiting time | 仿真纯 GPU vs 实测挂钟，latency 不受影响 |

4. **算法抗干扰排序**（对成本估算差异的鲁棒性，高负载 rps=0.125）：
   - p95_first (-0.92%) ≈ sjf_aging_guarded (+0.55%) ≈ p95_bucket_sjf_norm (+1.57%) > size_bucket_sjf_aging (-4.50%) > p95_bucket_sjf (+4.76%) > sjf_aging_no_preempt (+7.14%) > sjf_aging (+9.19%)
   - 在线学习类算法（p95_first、p95_bucket_sjf_norm）因学习收敛后不依赖初始 cost 估算，对齐最好
   - 频繁重新评估（sjf_aging_guarded, budget=5）能快速修正排序偏差
   - 无抢占算法（sjf_aging_no_preempt）因无法中途修正，排序匹配最低
